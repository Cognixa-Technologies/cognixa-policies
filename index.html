Security Policy for Developing

Section 1: Contribution

  • All contributions must be made as pull requests. These will only be accepted after a full security review has been conducted on the files. No exceptions. All changes must be traceable and reversible.
  • Intentionally malicious code will result in a blame link and immediate removal of the contributor from the project. A warning will be issued to all involved parties to prevent further security breaches.
  • Backups of your work are mandatory and should be stored securely under a form of encryption [1] provided by the Security team. NEVER store unencrypted backups on public or shared drives.
  • Ensure that all Environment Variables are encrypted using AES-GCM, with the shared key for Environment Variables being used strictly for this purpose. Any deviations will result in immediate review.
  • Contributed code must include full explanations of changes. Additionally, every change should be accompanied by a reference name in the format Initials-Short Date-Version (For example OA-5.11.2024-0.1.0) for clear tracking and versioning.
  • Code contributions must also be accompanied by tests where applicable. Testing is mandatory for any new feature, change, or patch.

Section 2: Writing

  • All code must pass a security review before being published, as outlined in Section 1, Point 1. Code that has not been reviewed will be immediately rejected.
  • Writers are expected to debug their code thoroughly. Automated debugging tools are allowed, but manual debugging is also a requirement to ensure functionality and security.
  • Methods such as exec() and eval() are NEVER to be used with user input, as they introduce significant security vulnerabilities. Always sanitize and validate input before processing.
  • Context Handlers, Decorators, Functions, and Type Checkers must be used whenever possible to limit the risk of errors. These tools help enforce rules, reduce redundancy, and clarify the purpose of code blocks. Follow the DRY (Don't Repeat Yourself) principle for cleaner, more secure code.
  • Strict adherence to PEP 8 or equivalent style guides for your language is expected. Maintain readability, consistency, and clarity in all code you write. This reduces the potential for introducing errors and security flaws.

Section 3: Test Hosting

  • Test hosting instances of Cognixa Core should only be done when the main instance is operational, or on your own private bot instance. Avoid using the main production instance unless absolutely necessary. Public exposure of testing environments is prohibited.
  • Data transmission over public networks should be avoided whenever possible. Use local environments, VPNs, or secure tunnels for communication. Ensure all data transfers are encrypted end-to-end.
  • For testing, if utilizing a Dual-Wrapper [3] for the bot, make sure to secure all exposed ports and disable them when they are not in active use. Leaving ports exposed unnecessarily is a significant security risk. Only expose necessary ports to the minimum required scope and disable unused services.
  • Testing scripts must be reviewed by a security team member before running in any shared or production environment. All sensitive data used in testing should be anonymized, and personally identifiable information should never be included in test data.
  • Before conducting tests on live systems, ensure all third-party services, dependencies, and libraries have been vetted for vulnerabilities. Use dependency scanning tools to check for known issues.

Section 4: Security Best Practices

  • Always follow secure coding principles, including proper input validation, output encoding, and sanitization. Remember that security starts with code quality and continues throughout the development lifecycle.
  • Follow the principle of least privilege for any processes, services, or users involved in development, testing, and deployment. Ensure only the minimum necessary permissions are granted to perform tasks.
  • Regularly audit and update your dependencies. Outdated or unpatched libraries are a common vector for exploits.
  • Encrypt all sensitive data in transit and at rest. Use industry-standard algorithms like AES-GCM for encryption and RSA for key exchange. Do not roll your own encryption solutions.
  • Ensure robust logging and monitoring of production systems. Set up alerts for suspicious activity or performance degradation that could indicate a potential security issue.
  • Regularly conduct security reviews, code audits, and penetration tests on all systems. Make security a continuous process, not just an afterthought.
